# Write Diagrams And Visualizations

`SKILL.md`의 실질적인 지침이 바뀌면 이 문서도 함께 갱신합니다.

## 이 스킬을 쓰는 경우

플로우차트, 표, 그래프, Mermaid, 이미지 등 시각화 도구를 Markdown에서 쓸 때 적용합니다.
이 스킬은 시각화의 공통 규칙을 정하고, 세부 상황별로 어느 스킬을 봐야 하는지 안내합니다.

## 공통 다이어그램 계약

- 한 다이어그램 안에서 화살표는 한 종류의 의미만 가지게 합니다.
  예: 수학 비중이 큰 THEORY 문서의 전체상에서는 화살표를 inclusion map이나 forgetful map 중 하나로만 둡니다.
- 별도 지침이 이미 정해져 있지 않다면, 먼저 화살표가 뜻하는 의미를 하나 정하고 다이어그램 아래에 그 의미를 적습니다. 여러 의미의 화살표가 필요하면 한 그림에 섞지 말고, 의미별로 별도의 다이어그램을 그립니다.
- `A -> B -> C` 같은 추이적 연결과 `A -> C` 같은 직접 연결을 한 다이어그램 안에 함께 그리는 것은 사용자의 허가 없이는 하지 않습니다.
- 추이적 연결과 직접 연결이 둘 다 필요해 보이면, 먼저 다이어그램이 가리키는 대상 간 연결 설계가 잘못된 것인지, 아니면 다이어그램으로의 표현이 잘못된 것인지 멈추고 검토합니다.
- 사용자가 허가한 경우에도, 추이적 연결을 보여 주는 그림과 직접 연결을 보여 주는 그림을 별도의 다이어그램 블록으로 나누는 쪽을 먼저 검토합니다.
- 구조가 복잡해져 가독성이 떨어지면, 한 그림에 억지로 넣지 말고 다이어그램을 나눕니다.
  다이어그램을 나눴다면, 그 둘 사이의 연결고리는 아래 설명에서 따로 적거나 새 다이어그램의 시작점을 앞 다이어그램에 이미 나온 적절한 블록으로 잡아 이어지는 지점을 쉽게 찾게 합니다.

## 공통 Mermaid 작성 규칙

- Mermaid는 language가 정확히 `mermaid`인 fenced code block으로만 씁니다.
- 들여쓰기만 한 Mermaid 텍스트, Mermaid 텍스트를 raw HTML로 감싸는 방식, `mermaid` fenced block이 아닌 다른 문법은 쓰지 않습니다.

## 세부 규칙을 볼 스킬

- 사이트 렌더 기준과 `RAW/**/*.md`용 Mermaid 점검은 `check-site-content`를 봅니다.
- Mermaid 라벨 안의 수학 비슷한 표기나 다이어그램 주변의 수학 표기 선택은 `write-math-notation`을 봅니다.
- THEORY 문서의 전체상, 분기 포인트, 문서 로드맵, 증명 로드맵은 `write-raw-content-theory`를 봅니다.
- `README.md`와 `docs/**/*.md`의 프로젝트 문서용 다이어그램은 이 스킬 자체를 봅니다.
